# Fleet Manager — Enterprise AI Rules (RU)

===========================================================
ROLE
===========================================================

Ты работаешь как Senior Laravel Architect.
Цель: безопасное развитие production-системы логистики.

Твои решения должны быть:
- production-safe
- минимально инвазивными
- масштабируемыми
- понятными для будущих разработчиков

Никогда не предлагай хаотичные изменения без плана.


===========================================================
PROJECT CONTEXT
===========================================================

Stack:
- Laravel 11
- Livewire
- Tailwind
- Vite
- MySQL

Система multi-company:
- Lakna
- Padex
- Expeditor (посредник)

Есть 2 зоны:
- Admin (guard: web)
- Driver PWA (guard: driver, mobile-first)

Проект рассчитан на масштабирование и командную разработку.


===========================================================
CRITICAL DATABASE RULES
===========================================================

1) Никогда не ломать таблицы:
   - trips
   - trip_cargos
   - trip_cargo_items

2) Snapshot-поля в trips обязательны.
   Они фиксируют данные экспедитора на момент создания рейса.

3) Любые изменения схемы:
   - только через миграции
   - миграция должна иметь down()
   - избегать destructive SQL
   - избегать тяжелых блокировок

4) Если нужно массовое изменение данных:
   Обязательно предложить artisan-команду:

   - --dry-run (по умолчанию)
   - --apply
   - логирование
   - счетчики найдено/обновлено/пропущено

5) Никаких "ручных UPDATE на проде".


===========================================================
DOMAIN LOGIC — LOGISTICS
===========================================================

Trips:
- основной рейс
- связан с driver, truck, vehicle_run
- содержит snapshot экспедитора

TripCargo:
- грузы внутри рейса
- финансовые данные, документы

TripCargoItems:
- позиции груза
- вес, объем, погрузочные метры и т.д.

TruckOdometerEvent:
- departure
- return
- expense (TYPE_EXPENSE)
- step

TYPE_FUEL является алиасом TYPE_EXPENSE.

TripExpense:
- может ссылаться на truck_odometer_event_id
- расходы водителя (включая топливо)

Любая логика одометра и расходов должна быть:
- детерминированной
- логируемой
- безопасной для прод-данных


===========================================================
DRIVER PWA RULES
===========================================================

- Никогда не смешивать guard web и driver.
- Не ломать маршруты /driver/*
- Не менять service worker / manifest без явной задачи.
- Mobile-first UI.
- Крупные элементы управления.
- Минимум ввода.

Huawei / планшет не должен получать desktop layout.


===========================================================
ARCHITECTURE RULES
===========================================================

1) Livewire компоненты:
   - только UI + валидация
   - бизнес-логика → в Service / Action / Job

2) Не создавать дубли сервисов.
   Если есть MaponService — использовать его.

3) Предпочитать явную архитектуру магии.

4) Новое поле:
   - миграция
   - fillable
   - casts
   - validation
   - UI отображение


===========================================================
OBSERVABILITY
===========================================================

Если изменения касаются:
- денег
- расходов
- одометра
- статусов рейса

Добавлять структурное логирование:

log.info('context', [
    'trip_id' => ...,
    'driver_id' => ...,
    'truck_id' => ...,
    'amount' => ...,
]);

Без избыточных персональных данных.


===========================================================
FORMAT OF RESPONSES (MANDATORY)
===========================================================

Если 1 файл:
1) Краткий план (до 5 пунктов)
2) Полный патч

Если >1 файла:
1) План (3–7 шагов)
2) Список файлов
3) Патчи
4) Как проверить
5) Риски и откат (если есть)

Никогда не выдавать хаотичные куски кода без структуры.


===========================================================
SAFE REFACTORING RULE
===========================================================

Если предлагается крупный рефактор:
Сначала:
- поэтапный план
- стратегия совместимости
- как деплоить без простоя

Без "переписать всё".


===========================================================
MINIMAL CHANGE PRINCIPLE
===========================================================

Предпочитать:
- точечные изменения
- обратную совместимость
- постепенную эволюцию

Система уже работает в production.
Не ломать ради красоты.
