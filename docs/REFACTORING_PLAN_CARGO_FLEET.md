# План синхронного рефакторинга: Cargo Trans & Fleet Manager

Оба проекта — параллельные копии для разных клиентов (разный бренд). Истины нет ни в одном. Цель: найти критические уязвимости и слабые места в обоих и вносить исправления **синхронно** в оба репозитория.

---

## Сводка по проектам

| Проект        | Критично | Высокий | Средний |
|---------------|----------|---------|---------|
| Fleet Manager | 2        | 6       | 10      |
| Cargo Trans   | 4        | 5       | 11+     |

Ниже — объединённый список проблем с указанием, где они есть (**F** = Fleet, **C** = Cargo Trans), и общая рекомендация по исправлению для обоих.

## 1. Безопасность

### Критические

| # | Проблема | F | C | Рекомендация (применить в обоих) |
|---|----------|---|---|-----------------------------------|
| 1.1 | **IDOR: водитель может открыть любой рейс** — в Driver PWA по ID рейса открывается любой рейс, без проверки `trip->driver_id === current driver` | ✓ | ✓ | В `TripDetails::mount()` после загрузки рейса: `abort_if($trip->driver_id !== Auth::guard('driver')->user()->driver->id, 403);` |
| 1.2 | **IDOR: водитель может открыть любой документ рейса** — по ID документа открывается документ любого рейса | ✓ | ✓ | В `ViewDocument::mount()`: проверить `$document->trip->driver_id === current driver->id`; иначе `abort(403)`. |
| 1.3 | **IDOR: генерация CMR/инвойса по cargo ID** — любой авторизованный админ может сгенерировать CMR/инвойс по чужому cargo | — | ✓ | В `CmrController::generateAndSave` / `generateInvoiceAndSave`: авторизовать доступ к cargo (например, `$cargo->trip->carrier_company_id === auth()->user()->company_id` или Policy). |
| 1.4 | **IDOR: открытие PDF инвойса и сохранение платежей** — любой админ может открыть любой инвойс и добавить платёж по ID | ✓ | ✓ | Scope по компании: при открытии инвойса и в `InvoicesTable::savePayment()` проверять, что инвойс принадлежит компании пользователя (через trip/carrier_company_id или Policy). |
| 1.5 | **Список инвойсов не ограничен по компании** — отображаются инвойсы всех компаний | — | ✓ | В `InvoicesTable::render()` ограничить запрос: `Invoice::query()->whereHas('trip', fn($q) => $q->where('carrier_company_id', auth()->user()->company_id))` (или общий scope). |
| 1.6 | **Dev-маршрут `/_dev/find-odometer`** — раскрытие структуры файлов; в Cargo — без auth | ✓ | ✓ | Удалить маршрут в прод или обернуть в `App::environment('local')` и `abort(404)` в production; в Cargo дополнительно повесить auth. |

### Высокий приоритет

| # | Проблема | F | C | Рекомендация |

| 1.7 | Нет Policy-based авторизации: доступ к ресурсам проверяется точечно, легко забыть проверку | ✓ | ✓ | Ввести Policies (Trip, TripCargo, Invoice, TripDocument, TripStepDocument) и использовать `$this->authorize()` в Livewire/контроллерах, `@can` в Blade. |
| 1.8 | Удаление документов/расходов по ID без проверки принадлежности к текущему рейсу/компании | ✓ | ✓ | В методах `deleteStepDocument`, `delete` (документы/расходы): после `findOrFail` проверять `$doc->trip_id === $this->trip->id` (или через step->trip_id) и для расходов `$exp->trip_id === $this->trip->id`; иначе 403. |
| 1.9 | Загрузка файлов: только `required|file`, без mime/size — риск произвольных типов и размера | ✓ | ✓ | Добавить правила, например: `'file' => 'required|file|mimes:pdf,jpeg,jpg,png,gif|max:10240'`, и по необходимости ограничить типы в ImageCompress/хранилище. |
| 1.10 | Жёстко прописан email для отчётов об ошибках (`rvr@arguss.lv`) | ✓ | ✓ | Вынести в config/env: `config('mail.error_report_to')` или `env('ERROR_REPORT_EMAIL')`, описать в `.env.example`. |
| 1.11 | В `.env.example` может быть `APP_DEBUG=true` — риск включённого отладки в проде | ✓ | ? | В `.env.example` указать `APP_DEBUG=false` и комментарий, что в production обязательно false. |

## Порядок синхронных обновлений

Рекомендуемая очерёдность (каждый пункт применять в **обоих** проектах):

1. **Критические IDOR и dev-маршрут**  
   - 1.1, 1.2: проверка водителя в TripDetails и ViewDocument.  
   - 1.3, 1.4, 1.5: авторизация CMR/инвойс, открытие PDF, список и сохранение платежей по компании (Cargo — все три, Fleet — где есть аналог).  
   - 1.6: убрать или защитить `/_dev/find-odometer`.

2. **Авторизация и загрузки**  
   - 1.7: ввести Policies и использовать их везде, где доступ к Trip/Invoice/Document.  
   - 1.8: проверка принадлежности при удалении документов и расходов.  
   - 1.9: ужесточить валидацию загрузки файлов.

3. **Конфиг и окружение**  
   - 1.10: email отчётов в env/config.  
   - 1.11: APP_DEBUG=false в .env.example.

4. **Данные и миграции**  
   - 3.1, 3.2, 3.3: роль, company IDs, nullable FK.  
   - 3.4, 3.5: логирование в MigrationHelper, имя файла миграции.

5. **Ошибки и UX**  
   - 4.1: не репортить ValidationException по умолчанию.  
   - 4.2, 4.3: обратная связь при no-op и в catch.

6. **Архитектура (по мере сил)**  
   - 2.1–2.3: сервисы, общие scope/policy, Form Request.

После каждого блока — проверка в обоих проектах (ручные сценарии и по возможности автотесты на критические пути).
