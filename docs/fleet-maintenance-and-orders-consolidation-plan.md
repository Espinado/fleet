# Обслуживание парка (Fleet Maintenance) и консолидация заказов — текущее состояние и план внедрения

---

## Часть 1. Несколько заказов в одном рейсе (консолидация)

**Правило:** Один заказ может быть привязан только к одному рейсу. При добавлении заказа в рейс у заказа выставляются `trip_id` и статус CONVERTED; повторное добавление того же заказа в другой рейс запрещено (проверка `trip_id` на карточке заказа и в `addOrderToTrip`). Реализовано в **Fleet** и **Cargo Trans**.

### Что уже есть в коде

| Компонент | Статус | Детали |
|-----------|--------|--------|
| **Модель Trip** | ✅ | `transportOrders()` — `hasMany(TransportOrder)`. `transportOrder()` — `hasOne()->oldest('id')` для обратной совместимости. |
| **Модель TransportOrder** | ✅ | Поле `trip_id` (nullable). Несколько заказов могут ссылаться на один рейс. |
| **Таблица trip_cargos** | ✅ | Колонка `transport_order_id` (nullable) — привязка груза к заказу-источнику. Миграция `2026_03_16_120000_add_transport_order_id_to_trip_cargos_table.php`. |
| **CreateTrip** | ✅ | Поддержка нескольких заказов: `from_order_id`, `from_order_ids`, `prefillFromOrderIds()`, `setOrdersForTrip()`. При сохранении всем выбранным заказам проставляются `trip_id` и статус CONVERTED. |
| **ViewTrip** | ✅ | Список заказов рейса (`$trip->transportOrders`). Кнопка «Добавить заказы в рейс» — выбор из пула (confirmed, без trip_id), вызов `addOrdersToTrip()`. У каждого заказа — «Отвязать заказ» (`removeOrderFromTrip()`). |
| **Логика добавления заказа в рейс** | ✅ | `appendOrderStepsAndCargosToTrip()` — создаёт TripStep и TripCargo из OrderStep/OrderCargo, проставляет `transport_order_id` у грузов. |
| **Синхронизация** | ✅ | `syncMissingCargosFromOrders()` — подтягивает недостающие грузы/шаги по заказам рейса. |

### Чего не хватает (по документу tms-gap-analysis.md)

| Функция | Описание | Сложность |
|---------|----------|-----------|
| **«Добавить в существующий рейс» со стороны заказа** | На карточке заказа (ShowOrder) при статусе confirmed и пустом trip_id — кроме кнопки «Создать рейс» — кнопка **«Добавить в существующий рейс»**: выбор рейса из списка (например, плановые/в работе на период), при выборе у заказа выставляются trip_id и converted. | Низкая |
| **Проверка использования transportOrder()** | В коде заменить использование `$trip->transportOrder` на `$trip->transportOrders` там, где ожидается несколько заказов (например, инвойсы, отчёты). | Низкая |

### План по консолидации (кратко)

1. **Карточка заказа (ShowOrder)**  
   Добавить кнопку «Добавить в существующий рейс» (видна при `status === confirmed` и `!trip_id`). По клику — модальное окно или отдельная страница с выбором рейса (фильтр: статус planned/in_progress, даты в нужном диапазоне). После выбора: обновить заказ (`trip_id`, status CONVERTED), вызвать логику добавления шагов/грузов в рейс (аналогично `appendOrderStepsAndCargosToTrip`), можно вынести в сервис/метод, вызываемый и из ViewTrip, и из нового экшена в ShowOrder.

2. **Аудит кода**  
   Найти все места, где используется `$trip->transportOrder` (единственное число). Если логика должна учитывать все заказы рейса — перейти на `$trip->transportOrders`.

3. **Инвойсы/отчёты**  
   При формировании счетов по рейсу учитывать, что грузы могут относиться к разным заказам (`transport_order_id`), при необходимости группировать по заказам.

## Часть 2. Обслуживание парка (Fleet Maintenance)

**Область:** Обслуживание (ремонты, ТО, напоминания, TCO) относится и к **тягачам (trucks)**, и к **прицепам (trailers)**. План и реализация — единые для **Fleet** и **Cargo Trans**. Часть полей изначально может быть пустой — данные заполняются по мере появления (например, пробег прицепа, интервалы ТО).

### Что уже есть

| Компонент | Статус | Детали |
|-----------|--------|--------|
| **Документы тягачей (Truck)** | ✅ | Техосмотр: `inspection_issued`, `inspection_expired`. Страховка: `insurance_*`, `insurance_expired`. Техпаспорт: `tech_passport_*`, `tech_passport_expired`. Лицензия: `license_*`, `license_expired`. |
| **Документы прицепов (Trailer)** | ✅ | Аналогично + TIR: `tir_issued`, `tir_expired`. |
| **Напоминания по документам** | ✅ | В OwnerDashboard — блок «Истекающие документы» (водители, тягачи, прицепы) на 30 дней. Есть задача/команда SendExpiringDocsNotifications (упоминалась в логах). |
| **Одометр** | ✅ | TruckOdometerEvent, интеграция Mapon — пробег по рейсам/периодам. Можно использовать для расчёта «следующее ТО по пробегу». |

### Чего нет (по документу)

| Функция | Описание | Сложность |
|---------|----------|-----------|
| **Учёт ремонтов и ТО** | Заявки на ремонт, фактические ТО по тягачу/прицепу (дата, пробег, вид работ, исполнитель, сумма). | Средняя |
| **Напоминания по ТО** | Следующее ТО по пробегу или по дате (интервал км или месяцев). | Низкая |
| **TCO по технике** | Суммарная стоимость владения: расходы (топливо и т.д.) + ремонты/ТО + по желанию амортизация. | Средняя |

### Предлагаемая модель данных (обслуживание парка)

**1. Таблица `vehicle_maintenance` (ремонты и ТО)**

Универсальная для **тягачей и прицепов** (полиморф: `maintainable_type` = Truck или Trailer). Часть полей допускается оставлять пустыми, пока данных нет (пробег прицепа, стоимость, исполнитель и т.д.).

| Поле | Тип | Назначение |
|------|-----|------------|
| id | bigint | PK |
| maintainable_type | string | `App\Models\Truck` или `App\Models\Trailer` |
| maintainable_id | bigint | ID тягача или прицепа |
| type | enum/string | `repair` \| `maintenance` \| `inspection` (или отдельная типология) |
| description | string, nullable | Описание работ |
| performed_at | date | Дата выполнения |
| odometer_km | int, nullable | Пробег на момент проведения (для прицепов может быть пустым) |
| cost | decimal, nullable | Стоимость (EUR), можно заполнять позже |
| vendor | string, nullable | Исполнитель / СТО |
| notes | text, nullable | Заметки |
| company_id | bigint, nullable | Для мультикомпании |
| created_at, updated_at | | |

**2. Поля в `trucks` и `trailers` для напоминаний по ТО**

Все поля nullable — заполняются по мере появления данных. Для прицепов пробег может отсутствовать, тогда используются только даты.

| Поле | Тип | Назначение |
|------|-----|------------|
| next_service_km | int, nullable | Следующее ТО по пробегу (км) |
| next_service_date | date, nullable | Следующее ТО по дате (если по регламенту раз в N месяцев) |
| service_interval_km | int, nullable | Интервал ТО по пробегу (км), например 30 000 |
| service_interval_months | int, nullable | Интервал ТО по времени (месяцев), например 12 |

После каждого ТО можно вручную или по правилу обновлять `next_service_km = odometer_km + service_interval_km` (и аналогично по дате). Если интервалы пока не заданы — напоминания по дате/пробегу не строятся до их заполнения.

**3. Напоминания по ТО**

- В дашборде (OwnerDashboard или отдельный блок «Предстоящее ТО»): список техники, у которой `next_service_km` не больше текущего пробега + порог (например 1000 км) или `next_service_date` в пределах 30 дней.
- Текущий пробег тягача — из одометра (последнее событие или сумма по рейсам за период). Для прицепов — либо только по дате, либо отдельное хранение пробега прицепа, если есть данные.

**4. TCO (Total Cost of Ownership)**

- По тягачу/прицепу за период: сумма `TripExpense` (топливо, дороги и т.д.) по рейсам этой техники + сумма `vehicle_maintenance.cost` по записям этой техники.
- Опционально: амортизация (норма * срок/пробег) — при наличии полей или правил в системе.
- Вывод: отдельная страница/отчёт «TCO по технике» или блок в карточке тягача/прицепа.

### План внедрения (обслуживание парка)

**Этап 1 — напоминания по ТО (низкая сложность)**  
1. Миграция: добавить в `trucks` и `trailers` поля `next_service_km`, `next_service_date`, `service_interval_km`, `service_interval_months`.  
2. Формы Edit Truck / Edit Trailer: поля «Следующее ТО (км)», «Следующее ТО (дата)», «Интервал ТО (км)», «Интервал ТО (месяцев)».  
3. OwnerDashboard (или отдельный виджет): блок «Предстоящее ТО» — техника, у которой скоро ТО по км или по дате (порог, например 30 дней / 2000 км). Использовать текущий пробег из одометра/рейсов.

**Этап 2 — учёт ремонтов и ТО (средняя сложность)**  
1. Миграция: создать таблицу `vehicle_maintenance` (полиморф или truck_id + trailer_id nullable).  
2. Модель VehicleMaintenance, связи в Truck и Trailer (`maintenanceRecords()` или аналогично).  
3. CRUD: список записей по тягачу/прицепу на карточке техники + форма создания/редактирования (дата, пробег, тип, описание, стоимость, исполнитель).  
4. При сохранении записи ТО с указанием пробега — опционально обновлять `next_service_km` у техники (например next = текущий пробег + interval).

**Этап 3 — TCO (средняя сложность)**  
1. Методы в Truck/Trailer: сумма расходов по рейсам за период (TripExpense по trip_id, где trip.truck_id/trailer_id = this) + сумма vehicle_maintenance.cost за период.  
2. Страница или блок «Стоимость владения» по выбранной технике за год/период: таблица или график (расходы по категориям + ремонты/ТО).  
3. При необходимости — экспорт (Excel/CSV) для отчётности.

## Итог

- **Консолидация заказов**: модель и основной сценарий (несколько заказов в одном рейсе, добавление/отвязка на ViewTrip, создание рейса из нескольких заказов в CreateTrip) уже реализованы. Остаётся добавить действие «Добавить в существующий рейс» на карточке заказа и при необходимости пройтись по коду для учёта всех заказов рейса.
- **Обслуживание парка**: есть только учёт документов и напоминания по их истечению. Для полноценного Fleet Maintenance нужно ввести учёт ремонтов/ТО (таблица + CRUD), поля «следующее ТО» и интервалы в trucks/trailers, напоминания по ТО, затем TCO по технике.

После выбора приоритета (сначала консолидация или сначала обслуживание парка) можно расписать конкретные шаги по файлам и задачам.
