# Концептуальный план реализации статистики

План размещения и реализации метрик по разделам приложения: **Счета**, **Статистика**, **Просмотр тягача**.

---

## 1. Дебиторская задолженность → раздел «Счета» (Invoices)

### Где
Страница списка счетов: `InvoicesTable` (`/invoices`).

### Что показывать
- **Блок-сводка** над таблицей (или под фильтрами), в одной строке с карточками:
  - **Сумма дебиторской задолженности** — сумма остатков по всем счетам с учётом фильтра по компании (т.е. `SUM(invoices.total - paid_total)` по тем же инвойсам, что видит пользователь). Вариант: считать только по статусам «не оплачен» и «частично оплачен».
  - **Количество счетов с задолженностью** — сколько инвойсов с остатком > 0.
  - **Просроченные счета** — количество счетов, у которых `due_date < today` и остаток > 0; опционально — сумма просроченных остатков.

### Источник данных
- Та же выборка, что и для таблицы: `Invoice` с `whereHas('trip', ... carrier_company_id)` по компании пользователя.
- Дополнительно: подзапросы или один запрос с `selectSub` для `paid_total` (как уже сделано в таблице).
- Агрегаты: один раз посчитать по всей выборке (без пагинации) суммы и счётчики.

### Реализация (концепт)
- В **InvoicesTable**: добавить вычисление сводки (например, в `render()` перед пагинацией): отдельный запрос или тот же базовый `Invoice::query()` с `whereHas('trip', ...)`, но без `paginate()` — взять `get()` и посчитать:
  - `total_receivables` = сумма `(total - paid_total)` по тем, где `paid_total < total`;
  - `invoices_with_balance_count`;
  - `overdue_count` и при желании `overdue_amount` (где `due_date < today` и остаток > 0).
- Передать в view переменные типа `$receivablesSummary` (объект или массив).
- Во **view** `invoices-table.blade.php`: после блока фильтров, перед таблицей — секция с 2–3 карточками (например в `grid grid-cols-3`): «Дебиторская задолженность», «Счета с остатком», «Просрочено».
- Учитывать валюту: если счета в разных валютах, можно показывать в основной валюте или с пометкой «EUR» как сейчас в таблице.

### Поведение фильтров
- Сводка по дебиторке может быть **глобальной** (все счета компании без учёта поиска/статуса), чтобы всегда видеть общую картину.
- Либо привязать к фильтру «статус»: при выборе «не оплачен»/«частично» показывать сводку только по отфильтрованным (тогда сводка пересчитывается при смене фильтра). Рекомендация: **глобальная** сводка по компании, чтобы не путать с числом строк в таблице.

## 2. Выручка / прибыль по клиенту → раздел «Статистика»

### Где
Новая подстраница в меню «Статистика»: **Статистика → По клиентам** (например маршрут `/stats/clients`).

### Что показывать
- Таблица: **клиент** (название, при желании ссылка на карточку), **количество рейсов** (или грузов), **сумма фрахта**, опционально **прибыль**.
- Фильтр по **периоду** (дата от / до), как на «Статистика → Обзор» (и быстрые кнопки 7d / 30d / 90d).
- Сортировка по сумме фрахта или по количеству рейсов.
- Итоговая строка: всего рейсов, всего фрахт.

### Источник данных
- **Выручка по клиенту**: `TripCargo` — группировка по `customer_id`, сумма `price`, количество грузов (или рейсов: считаем уникальные `trip_id`).
- Рейсы и грузы должны быть в рамках тех же ограничений по компании, что и в TripsStatsTable (global scope по `carrier_company_id`).
- **Прибыль по клиенту**: расходы привязаны к рейсу (`TripExpense`), а не к грузу. Варианты:
  - **Упрощённый**: показывать только **фрахт по клиенту** и **количество рейсов/грузов**; без прибыли по клиенту.
  - **Приближённый**: распределять прибыль рейса по клиентам пропорционально фрахту (доля фрахта клиента в общем фрахте рейса = доля прибыли). Тогда для каждого рейса считаем `trip_profit`, затем по каждому грузу в рейсе: `cargo_share = cargo.price / trip_freight`, `client_profit += trip_profit * cargo_share`. Реализационно сложнее, но даёт «прибыль по клиенту».

Рекомендация: на первом этапе — **фрахт + количество рейсов/грузов**; позже при необходимости добавить приближённую прибыль.

### Реализация (концепт)
- Новый Livewire-компонент: `App\Livewire\Stats\ClientsStatsTable` (или `StatsByClient`).
- Маршрут: `Route::get('/stats/clients', ClientsStatsTable::class)->name('stats.clients');`
- В **layouts/app.blade.php** в блоке меню «Статистика» добавить пункт: «По клиентам» → `route('stats.clients')`.
- Компонент: свойства `dateFrom`, `dateTo`, `search` (по имени клиента), `sortField`, `sortDirection`. Запрос: из `TripCargo` с `join Trip` и фильтром по `Trip.carrier_company_id` (через пользователя) и по датам `Trip.start_date`. Группировка по `customer_id`, выборка `Client` по связи. Передача в view: коллекция с полями client_id, client_name, cargos_count, trips_count (unique trip_id), freight_total.
- View: таблица + фильтры по периоду (аналогично TripsStatsTable), итоговая строка.

## 3. Расходы по категориям → раздел «Статистика» (улучшение имеющейся)

### Где
- **Статистика → Обзор** (TripsStatsTable): добавить **новый блок** «Расходы по категориям» за тот же период, что и рейсы.
- **Статистика → События одометра** (EventsTable): оставить и при желании слегка улучшить уже имеющуюся сводку по категориям (например визуально выделить или добавить мини-график).

### Что показывать на Обзоре
- Один блок под сводкой по рейсам (или рядом): **таблица или набор карточек** по категориям расходов (`TripExpenseCategory`): название категории, сумма за период, доля в % от общей суммы расходов. Для топлива/AdBlue — при желании и литры.
- Период и компания — те же, что в фильтрах Обзора (dateFrom, dateTo и scope по компании через рейсы).

### Источник данных
- `TripExpense`: по рейсам пользователя (`Trip` с `carrier_company_id`), за период по полю `expense_date` (или по дате рейса `Trip.start_date` — лучше по дате рейса, чтобы совпадало с выборкой рейсов). Исключить категорию `SUBCONTRACTOR` при необходимости (как в EventsTable).
- Группировка: `category`, `SUM(amount)`, `SUM(liters)` (для fuel/adblue/washer_fluid).

### Реализация (концепт)
- В **TripsStatsTable** в `render()`: после получения выборки рейсов за период можно взять те же `dateFrom`/`dateTo` и по ним отфильтровать `TripExpense` через связь с `Trip` (company scope уже через Trip). Либо один запрос: `TripExpense::query()->whereHas('trip', fn...)`, whereBetween по дате рейса, groupBy('category'), select raw SUM(amount), SUM(liters). Результат передать в view как `$expensesByCategory`.
- Во **view** `trips-stats-table.blade.php`: после блока сводки (summary) и перед графиком — секция «Расходы по категориям»: список категорий с подписями из `TripExpenseCategory::options()` и суммами; итоговая строка «Всего».
- В **EventsTable** сводка по категориям уже есть; при желании — добавить небольшую визуализацию (например горизонтальные бары долей) или переставить блок выше, чтобы был заметнее.

## 4. Пробег по тягачу за период → раздел «Просмотр тягача»

### Где
Страница карточки тягача: `ShowTruck` (`/trucks/{truck}`).

### Что показывать
- Новый блок (секция) на странице, например после блока Mapon или после техосмотра: **«Пробег за период»**.
- Элементы:
  - Фильтр по периоду: **дата от** и **дата до** (и при желании быстрые кнопки: 30 дней, 90 дней, текущий месяц).
  - Показатели: **суммарный пробег (км)** за период, **количество рейсов**.
  - Опционально: таблица последних рейсов по этому тягачу (номер рейса, дата, пробег по рейсу) — для детализации.

### Источник данных
- Рейсы: `Trip::where('truck_id', $truck->id)`, фильтр по `start_date` (или по диапазону дат рейса) в выбранном периоде.
- Пробег по рейсу: как в TripsStatsTable — приоритет по событиям одометра (первое событие TYPE_DEPARTURE и последнее TYPE_RETURN по `trip_id`), иначе `odo_end_km - odo_start_km` из `trips`. Формула: для каждого рейса `distance_km = return_odometer - departure_odometer` (или `odo_end_km - odo_start_km`), затем сумма по всем рейсам за период.

### Реализация (концепт)
- В **ShowTruck** добавить свойства: `$periodFrom`, `$periodTo` (по умолчанию — последние 30 дней).
- В `render()` или через вычисляемое свойство: запрос рейсов по `truck_id` и датам; для каждого рейса вычислить пробег (подзапросы к `TruckOdometerEvent` или поля `odo_start_km`/`odo_end_km`). Альтернатива: вынести логику в отдельный метод `getTruckMileageStatsProperty()`: возвращает `['total_km' => ..., 'trips_count' => ..., 'trips' => [...]]`.
- Во **view** `show-truck.blade.php`: новая секция «Пробег за период» с полями ввода дат (wire:model), кнопками быстрого выбора периода, отображением `total_km` и `trips_count`, и при желании — компактной таблицей рейсов (ссылка на рейс, дата, км).
- Учитывать права доступа: пользователь может видеть тягача только своей компании — рейсы уже будут в рамках этой компании через scope Trip.

## Порядок внедрения

1. **Дебиторка в Invoices** — изолированное изменение, быстрый эффект для контроля платежей.
2. **Пробег в ShowTruck** — локально в карточке тягача, без новых маршрутов.
3. **Расходы по категориям в Обзоре** — расширение TripsStatsTable и его view.
4. **Статистика по клиентам** — новый компонент и пункт меню, при необходимости потом добавить «прибыль по клиенту» (пропорционально).

После этого можно обновить **stats-recommendations.md**, отметив реализованные пункты и при необходимости добавить cost per km в Обзор или в карточку тягача.
