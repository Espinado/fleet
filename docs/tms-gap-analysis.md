# Анализ пробелов: Fleet Manager / Cargo Trans vs полноценная TMS

Сравнение текущего функционала с типичными модулями и возможностями Transport Management System (TMS).

---

## Что уже есть (реализовано)

| Область | Реализовано |
|--------|--------------|
| **Управление рейсами** | Создание, редактирование, просмотр рейсов; экспедитор, перевозчик (свой/third party), транспорт (водитель, тягач, прицеп); маршрут (шаги погрузка/разгрузка), порядок шагов (редактор); статусы рейса. |
| **Грузы (cargos)** | Грузы по клиентам, привязка к шагам; позиции (описание, паллеты, вес, объём, погрузочные метры); контейнер/пломба, TIR/таможня; задержки (dikstave): дни и сумма. |
| **Документы** | CMR и Invoice по грузу (генерация PDF); документы по шагам и общие по рейсу (загрузка JPG); транспортный заказ (order). |
| **Расходы** | Расходы по рейсу (категории: топливо, AdBlue, дороги, парковка и т.д.); привязка к одометру; исключение SUBCONTRACTOR в сводной статистике. |
| **Одометр и события** | Выезд/возврат в гараж, события по шагам; интеграция Mapon (CAN/odometer); сводка по категориям в «Событиях». |
| **Водители** | CRUD, документы (права, Code 95, медсправка и т.д.), PIN для входа; приложение водителя (PWA): рейсы, статусы шагов, загрузка документов, расходы, выезд/заезд с одометром. |
| **Транспорт** | Тягачи и прицепы: CRUD, документы (техосмотр, страховка, техпаспорт); пробег за период (выезд/заезд), пагинация по рейсам; то же для прицепов. |
| **Клиенты** | CRUD компаний (клиенты, отправители, получатели); реквизиты, контакты. |
| **Счета и оплаты** | Список счетов, генерация из рейса; внесение платежей; дебиторская задолженность (сводка: сумма, кол-во с остатком, просрочено). |
| **Статистика** | Обзор: рейсы за период, фрахт, расходы, прибыль, маржа %, cost per km; расходы по категориям; события одометра; по клиентам (выручка, рейсы); простои (downtime); карта (Mapon). |
| **Контроль документов** | Dashboard: истекающие документы (водители, тягачи, прицепы, грузы) на 30 дней. |
| **Мультикомпания** | Несколько компаний-перевозчиков; разграничение по carrier_company_id; карта по доступу. |
| **Уведомления** | Push (VAPID), тестовая рассылка; опционально уведомления при входе водителя / выезде. |

## Чего не хватает для полноценной TMS

### 1. Заказы и планирование (Order Management)

| Функция | Описание | Сложность |
|--------|----------|-----------|
| **Заявки/заказы на перевозку** | Отдельная сущность «Заказ» до создания рейса: маршрут, объём, желаемые даты, клиент; статусы (черновик, подтверждён, отменён). | Средняя |
| **Пул заказов и планирование** | Сводный экран заказов; назначение на рейс/водителя; календарь или план-график рейсов. | Средняя |
| **Котировки (quotes)** | Предложение цены клиенту до подтверждения; версии, согласование. | Средняя |

Сейчас рейс создаётся сразу; нет этапа «заявка → рейс».

#### Как это может выглядеть в нашем случае (конкретный вариант)

Ниже — вариант, завязанный на ваши сущности (Company, Client, Trip, TripStep, TripCargo) и поток «экспедитор → перевозчик → маршрут → грузы».

**1. Сущность «Заказ на перевозку» (Transport Order)**

Отдельная сущность **до** создания рейса — «заявка» с маршрутом и грузом в упрощённом виде.

| Поле | Назначение |
|------|------------|
| **Номер, дата заявки** | Идентификация, порядок (например `TO-2025-001`). |
| **Экспедитор** | `expeditor_id` → Company (кто дал заказ; реквизиты можно копировать в рейс при создании). |
| **Заказчик по грузу** | `customer_id` → Client. |
| **Желаемые даты** | `requested_date_from`, `requested_date_to` или одна `requested_date` (ориентир для планирования). |
| **Маршрут заказа** | 2+ шага в отдельной таблице `order_steps`: тип (погрузка/разгрузка), страна, город, адрес, желаемые дата/время, контакт. По смыслу — как TripStep, но без привязки к рейсу. |
| **Груз по заказу** | В таблице `order_cargos`: отправитель/получатель (shipper_id, consignee_id → Client), вес, объём, паллеты, описание; при создании рейса это станет TripCargo. |
| **Цена** | `quoted_price`, `currency` — предложенная/согласованная цена (котировка). |
| **Статус** | `draft` → `quoted` → `confirmed` → `converted` | `cancelled`. |
| **Ссылка на рейс** | `trip_id` (nullable) — заполняется, когда заказ «превратили» в рейс. |
| **Заметки** | Внутренние и для клиента. |

Идея: заказ описывает **что и куда везём, когда и за сколько**, без привязки к водителю/тягачу/прицепу. Транспорт и детали рейса появляются уже при создании рейса из заказа.

**2. Экраны и сценарии**

- **Список заказов** (`/orders`)  
  Таблица: номер, дата заявки, экспедитор, заказчик, маршрут (откуда → куда), желаемая дата, цена, статус. Фильтры: по статусу (новые, предложено, подтверждённые, в работе, отменённые), по дате, по клиенту/экспедитору. Действия: создать, открыть, отредактировать, «Создать рейс из заказа».

- **Карточка заказа**  
  Вся информация по заказу: экспедитор, заказчик, маршрут (список шагов с адресами и датами), груз (вес, объём, отправитель/получатель), котировка, статус. Кнопки: «Изменить», «Предложить цену» (перевод в quoted), «Подтвердить» (confirmed), **«Создать рейс»** (главный переход в планирование).

- **Создание рейса из заказа**  
  По кнопке «Создать рейс» открывается текущий сценарий создания рейса (CreateTrip), но с **предзаполнением** из заказа: экспедитор, маршрут (шаги копируются в TripStep), груз (в TripCargo), желаемая дата → start_date. Менеджер достраивает: свой/third party, водитель, тягач, прицеп, при необходимости правит шаги и грузы. После сохранения рейса у заказа проставляются `trip_id` и статус `converted`.

- **Планирование (сводка)**  
  Один экран, где видно и заказы, и рейсы:
  - Блок «Заказы к планированию» — заказы со статусом `confirmed` и пустым `trip_id` (ещё не назначены на рейс).
  - Блок «Рейсы на период» — список/календарь рейсов (например на неделю) с датами и водителями.
  - Из заказа — одно действие: «Создать рейс» (как выше). При желании позже можно добавить «Добавить в существующий рейс» (один рейс — несколько заказов/грузов), но на первом этапе достаточно «один заказ → один новый рейс».

Так заказы и планирование в вашем случае выглядят как этап **«заявка → котировка → подтверждение → создание рейса»** без изменения текущей логики рейсов и грузов.

**3. Минимальный набор таблиц (миграции)**

- `transport_orders`: номер, дата, expeditor_id, customer_id, requested_date_from, requested_date_to, quoted_price, currency, status, trip_id (nullable), notes, timestamps.
- `order_steps`: order_id, type (loading/unloading), country_id, city_id, address, date, time, contact_phone, order (сортировка).
- `order_cargos`: order_id, customer_id, shipper_id, consignee_id, weight_kg, volume_m3, pallets, description, quoted_price (если несколько грузов в одном заказе).

**4. Ограничения и упрощения**

- Один заказ → один рейс (связь 1:1 по `trip_id`). Консолидация «несколько заказов в один рейс» — отдельный следующий шаг.
- Котировка — одно поле «цена» по заказу (или по каждому order_cargo). Отдельный модуль версий котировок и согласований при необходимости можно добавить позже.
- Рейс по-прежнему можно создавать **без** заказа (как сейчас); заказы — опциональный контур для планирования и учёта заявок.

В итоге: заказы и планирование в нашем случае — это **справочник заявок с маршрутом и грузом** и явный переход **«Создать рейс из заказа»** с предзаполнением, без дублирования логики рейсов.

**5. Грузы на этапе заказа: известные и неизвестные (как в крупных TMS)**

В крупных TMS заказ на перевозку часто допускает два сценария:

| Сценарий | Описание | Реализация у нас |
|----------|----------|-------------------|
| **Груз известен** | Клиент при заявке указывает позиции: отправитель/получатель, вес, объём, паллеты, описание; при создании рейса эти данные копируются в TripCargo. | В форме заказа (создание/редактирование) — блок «Грузы по заказу»: можно добавить 0, 1 или несколько строк (customer, shipper, consignee, description, weight_kg, volume_m3, pallets, quoted_price). Поля опциональны: минимум — описание или «уточняется». |
| **Груз неизвестен / уточняется** | Заказ принимается с маршрутом и датами; состав груза заполняют позже (при подтверждении или при создании рейса). | Те же формы: блок грузов может быть пустым. При «Создать рейс из заказа» в CreateTrip подставляются только шаги и общие данные; грузы менеджер добавляет вручную в форме рейса. |
| **Маршрут (шаги)** | Аналогично: точки погрузки/разгрузки могут быть известны по заявке или уточняться позже. | Блок «Маршрут заказа (шаги)» в форме заказа: динамический список шагов (тип loading/unloading, страна, город, адрес, дата, время, контакт). Пустой список допустим — маршрут задаётся при создании рейса. |

Итого: **груз и маршрут на заказе опциональны**. Если указаны — при «Создать рейс» они подставляются в форму рейса; если нет — рейс создаётся с пустыми шагами/грузами и заполняется вручную. Так достигается гибкость «как в больших TMS» без избыточной сложности.

**6. Несколько заказов в одном рейсе (консолидация) — концепция**

Сейчас: один заказ → один рейс (связь 1:1). В БД поле `transport_orders.trip_id` уже позволяет нескольким заказам ссылаться на один и тот же рейс; в коде у Trip стоит `hasOne(TransportOrder)` — для консолидации нужно перейти на **один рейс → много заказов**.

| Аспект | Как может выглядеть |
|--------|----------------------|
| **Модель данных** | Trip: `hasMany(TransportOrder::class)` вместо `hasOne`. Один и тот же `trip_id` у нескольких записей в `transport_orders`. Никаких новых таблиц. |
| **Создание рейса** | **Вариант А:** «Создать рейс из заказа» (как сейчас) — первый заказ даёт маршрут и грузы; после сохранения рейса в том же или другом месте интерфейса — «Добавить в этот рейс ещё заказы». **Вариант Б:** экран «Новый рейс» с шагом «Выбрать заказы» — выбор одного или нескольких подтверждённых заказов (без `trip_id`); маршрут и грузы объединяются из выбранных заказов (шаги склеиваются/дедуплицируются, грузы добавляются списком). |
| **Просмотр рейса** | Блок «Заказы в рейсе»: список заказов (номер, экспедитор, клиент, статус), у каждого `trip_id` = этот рейс. Кнопка «Добавить заказ в рейс» → выбор заказа из пула (confirmed, без trip_id) → у выбранного проставляются `trip_id` и статус `converted`. Для незавершённого рейса — возможность «Отвязать заказ» (очистить `trip_id`, вернуть статус в confirmed), если политика бизнеса это допускает. |
| **Просмотр заказа** | Для заказа со статусом «к планированию» (confirmed): кроме «Создать рейс» — кнопка **«Добавить в существующий рейс»**: выбор рейса из списка (например, плановые/в работе на период); при выборе у заказа выставляются `trip_id` и `converted`. |
| **Маршрут и грузы рейса** | Рейс по-прежнему хранит один общий маршрут (TripStep) и список грузов (TripCargo). При добавлении второго (и далее) заказа: либо **вручную** дополнять шаги и грузы в форме рейса (скопировав из заказа), либо **полуавтомат**: при «Добавить заказ в рейс» показывать дифф: «Из заказа N добавятся шаги: …, грузы: …» и кнопку «Включить в рейс» (аппенд шагов и грузов в текущий рейс). Историю «этот TripCargo из заказа X» при желании можно хранить в поле `transport_order_id` в `trip_cargos` (опционально, для отчётов и инвойсов). |
| **Инвойсы** | Остаётся логика «счёт по рейсу/по грузу»: можно строить инвойс по рейсу с разбивкой по клиентам/заказам (по одному или несколько заказов в рейсе). |
| **Ограничения** | Заказ со статусом `converted` не редактируется (как сейчас); изменения — только в рейсе. Добавлять в рейс можно только заказы в статусе confirmed и без `trip_id`. |

Кратко: **несколько заказов в один рейс** = разрешить у Trip связь «много заказов» + экраны «Добавить заказ в рейс» (со стороны рейса) и «Добавить в существующий рейс» (со стороны заказа), с объединением шагов/грузов вручную или полуавтоматом.

### 2. Оптимизация маршрутов и загрузки

| Функция | Описание | Сложность |
|--------|----------|-----------|
| **Оптимизация маршрута** | Расчёт порядка точек (TSP), учёт времени, ограничений; предложение «оптимального» порядка шагов. | Высокая |
| **Расчёт километража/времени** | Автоматический километраж и ETA по маршруту (интеграция с картами/API). **Помощник менеджера для расчёта тарифов:** предполагаемый километраж по маршруту даёт основу для котировки и фрахта (например, тариф €/км × расчётный км = ориентир суммы). | Средняя |
| **Консолидация грузов** | Объединение нескольких заказов в один рейс; подбор по направлениям и датам. | Высокая |

Сейчас маршрут задаётся вручную; километраж — из одометра или полей рейса. Расчётный километраж по маршруту (до выполнения рейса) позволил бы менеджеру обоснованно считать тарифы и котировки.

#### Где разместить и как реализовать

Цель: дать менеджеру **предполагаемый километраж** по маршруту (заказа или рейса), чтобы он мог рассчитать тариф/котировку (например, €/км × расчётный км = ориентир фрахта).

| Место | Зачем | Реализация |
|-------|--------|------------|
| **Заказ (Order)** | Котировка и фрахт чаще считают на этапе заказа. Маршрут уже есть в `order_steps` (погрузка/разгрузка с адресами). | **Карточка заказа (ShowOrder):** блок «Расчёт маршрута» с кнопкой «Рассчитать километраж». По нажатию — запрос к API маршрутизации по шагам заказа; вывести «Расчётный км: X XXX», опционально «Ориентир фрахта при тарифе ___ €/км: ___ EUR». При желании сохранять `estimated_km` в заказ для истории. **Создание/редактирование заказа (CreateOrder / EditOrder):** в блоке маршрута (order steps) — та же кнопка или автоматический расчёт при ≥2 точках; показывать расчётный км под списком шагов. |
| **Рейс (Trip)** | При создании/редактировании рейса маршрут задаётся в `trip_steps`. Расчётный км полезен для планирования и для сравнения с фактом (одометр). | **Создание/редактирование рейса (CreateTrip / EditTrip):** в секции «Маршрут (шаги)» — кнопка «Рассчитать километраж» или расчёт при изменении шагов; вывести «Расчётный км: X XXX». Опционально сохранять в рейс поле `estimated_km`. **Просмотр рейса (ViewTrip):** показывать «По маршруту: X км» и при наличии факта — «Факт (одометр): Y км». |

**Общая логика:**

1. **Сервис расчёта** (например `RouteDistanceService`): на вход — массив шагов с адресами (для каждого шага: страна + город + address из `OrderStep` или `TripStep`). Сборка строки адреса → запрос к API маршрутизации (OSRM, GraphHopper, Google Directions и т.п.) → возврат `distance_km`, опционально `duration_minutes` (ETA).
2. **Единый источник шагов:** для заказа — `$order->steps` (OrderStep), для рейса — `$trip->steps` (TripStep). Оба имеют `country_id`, `city_id`, `address` (и связь с Country/City для названий) — этого достаточно для построения адреса.
3. **UI:** в каждом из перечисленных экранов — блок или строка с кнопкой «Рассчитать километраж»; после ответа API — вывод результата. Опционально: поле «Тариф €/км» (ввод или из настроек) и расчёт «Ориентир фрахта = тариф × км».
4. **Хранение:** по желанию — поле `estimated_km` (и при необходимости `estimated_duration_minutes`) в `transport_orders` и в `trips`, чтобы не пересчитывать каждый раз и использовать в отчётах/котировках.

**Приоритет размещения:** в первую очередь **карточка заказа** и **форма создания/редактирования заказа** (там формируется котировка и цена); затем — форма создания/редактирования рейса и просмотр рейса (планирование и сравнение с фактом).

#### Какой API использовать

Нужны два этапа: **(1) геокодирование** адресов шагов (страна + город + address → lat, lon) и **(2) маршрут** по точкам (coordinates → distance_km, duration).

| Вариант | Геокодирование | Маршрут | Плюсы | Минусы |
|--------|----------------|---------|--------|--------|
| **OSRM + Nominatim** | Nominatim (OSM, бесплатно) | OSRM public (router.project-osrm.org) или свой сервер | Бесплатно, без ключей; OSRM — расстояние в метрах, время в секундах | Nominatim лимит ~1 запрос/сек; публичный OSRM — лимиты для продакшена; два разных сервиса |
| **OpenRouteService** | ORS Geocoding | ORS Directions | Один ключ, одна документация; ~2000 запросов/день бесплатно; Европа покрыта хорошо | Нужна регистрация (api.openrouteservice.org); общий лимит на все запросы |
| **GraphHopper** | GraphHopper Geocoding | GraphHopper Route | Удобно для логистики; есть бесплатный tier | Лимиты на бесплатном плане |

**Рекомендация:** для старта — **OpenRouteService** (один API key, и геокодирование, и маршрут, 2000 req/день). Если понадобится больше запросов или не хотите зависеть от ключа — **OSRM** (маршрут) + **Nominatim** (геокодирование) или свой инстанс OSRM.

**Формат запроса (OSRM Route):**  
`GET https://router.project-osrm.org/route/v1/driving/{lon1},{lat1};{lon2},{lat2};...?overview=false`  
Ответ: `routes[0].distance` (метры), `routes[0].duration` (секунды). Конвертация: `distance_km = distance / 1000`, `duration_minutes = duration / 60`.

**Формат запроса (OpenRouteService Directions):**  
POST с телом `{"coordinates": [[lon1,lat1],[lon2,lat2],...]}`; в ответе — `routes[0].summary.distance` (м), `routes[0].summary.duration` (с). Документация: https://openrouteservice.org/dev/#/api-docs/v2/directions/{profile}/json/post.

**Важно:** если в будущем при сохранении шага (OrderStep / TripStep) сохранять ещё и `lat`, `lon` (из геокодинга при вводе адреса), то при расчёте маршрута геокодировать заново не нужно — сразу передаём координаты в OSRM/ORS.

## Итог

Система уже покрывает **ядро TMS**: рейсы, маршруты, грузы, документы (CMR, счета), расходы, одометр, водительское приложение, счета и дебиторку, базовую статистику и карту. Для **полноценной TMS** в первую очередь не хватает:

1. **Заказов/заявок** до рейса и планирования.
2. **Отслеживания для клиента** (публичная страница/ссылка).
3. **Сводного дашборда владельца** (выручка, дебиторка, топ клиенты/тягачи, задержки).
4. **Управления перевозчиками** (справочник, тарифы).
5. **Обслуживания парка** (ремонты, ТО по пробегу).
6. **Экспорта отчётов** и при необходимости **API** для интеграций.

Документ можно использовать как дорожную карту развития в сторону полноценной TMS.
